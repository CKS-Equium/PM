# Quality Engineer — notes

Playbook for the Quality Engineer. This agent appends freely; entries are **not** gated.
Entry format: **situation → what happened → lesson → confidence**.
The Process Engineer periodically distills proven lessons here into `persona.md` (via reviewed PR).

---

## team-pulse-dashboard (2026-06-03)

- **Weak test let a real defect through** — *Situation:* my suite (46→55 tests, all ACs mapped) covered the multi-milestone/phase-conflict path. *What happened:* the multi-milestone test was too weak to catch the **false phase-conflict** badge (milestone derived from a closed/arbitrary issue); the Reviewer caught it at gate 5 instead. *Lesson:* for derived/aggregated state, test the *negative* and the noisy cases (closed issues, no open milestone, ambiguous data), not just the happy multi-item path — assert the resolved value, not just that branching occurred. *(confidence: high)*
- **Live contract tests coupled to an external server** — *Situation:* the contract tests hit an externally-running server rather than spinning up their own. *What happened:* tests passed locally but are flaky in CI because they assume a server is already up — not self-contained. *Lesson:* contract/integration tests must own their lifecycle (start a server fixture, bind ephemeral port, tear down); never depend on an externally-running process. Flag this as a refactor when inherited. *(confidence: high)*

## hearthflix (2026-06-04)

- **Test the resource-bound NFR, not just the feature** — *Situation:* I grew the suite to 49/49 at gate 4 with good functional coverage (added transcode ready-signal both branches, capacity reaping, watchdog, path-traversal). *What happened:* I tested that transcoding worked, but I did **not** assert the design's "disk is bounded per session" NFR — so the unbounded-`/data` bug (H1) slid past gate 4 to the Reviewer at gate 5. *Lesson:* when the design states a resource bound (disk/memory/process count), write a test that **asserts the bound holds** (e.g. session dir is empty/removed after stop), not just that the happy-path feature runs. Treat every NFR as a test obligation. *(confidence: high)*
- **Cover dependency-death and unbounded-wait paths** — *Situation:* the transcode entered a wait-for-ready state. *What happened:* "Preparing forever" when ffmpeg died (M1) was a wait-state defect; my added tests covered the ready signal but the death-during-wait case needed explicit coverage. *Lesson:* for any wait/poll state, add a negative test where the dependency dies/fails immediately, and assert the system reaches an error state within a bound — don't only test the success transition. *(confidence: high)*

## colonygame / Hiraeth (2026-06-08) — execution-mode autonomous build

- **Test the REAL shipped data, not a hand-authored fixture — every gate from the start** — *Situation:* a data-driven colony-sim where behaviour comes from `data/buildings.json`. *What happened:* all the T1 unit tests (38) ran against a **hand-authored fixture**, not the real shipped `data/buildings.json` — so a mis-authored `reach` that made the smelter silently never smelt was **invisible to the unit suite** and only the headless scenario caught it. The fix that proved out was a `ShippedEconomyTests` integration test that runs the **actual** `data/*.json` and asserts the economy closes (flip the kind → it fails red — proven teeth). *Lesson:* for any data-driven design, ship a **real-shipped-data integration test every gate from the start** that loads the production data files (not a fixture) and asserts the authored content actually works — fixture-only tests cannot catch authored-data breaks, which are exactly the breaks that ship. **Candidate build-DoD.** *(confidence: high)*
- **Assert USABILITY, not just presence/conservation** — *Situation:* the T4 logistics gate stored production goods in warehouses. *What happened:* the warehouse "roach motel" bug passed because the tests/scenario asserted only **conservation** (no goods lost) — but the goods were stuck in dead storage, never readable back, so production silently drained. The reviewer caught it; the fix was a round-trip test (15 stranded → 0). *Lesson:* for storage/transport/transform stages, conservation and presence are **not** acceptance criteria — assert the output is **consumed/usable downstream** (full produce→consume loop). Also avoid assertions that can pass at a **degenerate value** (the T1 satisfaction assertion silently passed at `avgSat=0`) — pin the assertion to a meaningful non-trivial value. *(confidence: high)*

## colonygame / Hiraeth (2026-06-08) — DEV_PLAN_2 player-first QoL (playtest-gated)

- **The headless test net verifies the SIM, not the GAME — green CI ≠ playable** — *Situation:* the player-first QoL phase where the gate was a human playtest; the `t0–t5` headless scenarios + the unit suite (200+→234) were the machine layer. *What happened:* the test net did its job perfectly **as a regression net** — across ~9 presentation-heavy PRs (incl. a fixed-timestep refactor + a `FounderTier`/`PlayerTier` reconciliation) the proven sim economy **never accidentally regressed once** (0 escaped sim defects). **But every single playtest failure was a UI/interaction defect** the net is structurally blind to: buttons rebuilt every frame (clicks lost), enable-state not mirroring sim preconditions (a lying button), input buried in the fixed-timestep loop (dropped clicks), FPS regressions, dev keys in the player build. The machine layer was a false "all green." *Lesson:* for player-facing work, read a green test net as **"the sim didn't regress," NOT "the game works."** The feel/UI/interaction layer is **structurally outside** the headless test net's reach — the only real gate there is human (playtest), backed by a pre-playtest interaction self-audit/review for the diff-visible GUI pitfalls. Keep the regression net (it makes fast surface iteration safe) but never let its green light be mistaken for a passing game. *(confidence: high)*

## new-cadair A/B experiment (2026-06-16) — autonomous Phase-A rebuild vs reference

- **Metric-span honesty: a green performance number can measure the wrong span** — *Situation:* comparing the live `scan_time_ms` the two builds emit on the bus. *What happened:* new-cadair's live `scan_time_ms` read **0.7 µs** and OG's read **0.73 ms** — a 1000× apparent gap that is purely an **artifact of what each times**: new-cadair times only the Rust `engine.scan()`; OG times the **whole orchestrator cycle** (adapter read + engine + publish). OG's is the operationally meaningful number ("can we hold the 100 ms loop end-to-end?"); new-cadair's flatters itself by hiding adapter + publish latency. Both are "green." *Lesson:* when a metric is used as a **budget/gate** (scan time, latency, throughput), the evidence/RTM must **declare the span it covers** and prefer the **end-to-end** span; a bench/engine-only figure must not be presented as the cycle figure. Add a **"metric honesty" sibling** to the failure-mode review lens: for every perf claim, ask *what is timed, and is that the span the requirement is about?* *(confidence: high)*
- **The system-validation gate must exercise the primary OPERATOR-INITIATED path, not just an auto-driven one** — *Situation:* new-cadair auto-runs its recipe phase driver on boot (so the overnight autonomous run needed no human), while OG requires an operator "Start Batch" (`POST /recipe/start`) to run. *What happened:* new-cadair's runnable demo + validation gate proved the **live data path** (sim→engine→bus→bridge→WS frame) end-to-end, but **never exercised the operator-initiated recipe-start path** the spec implies a real operator takes — that path was untested by the gate. *Lesson:* a validation gate that proves the data path is necessary-but-insufficient if the product has a **primary operator entry action**; the gate (or acceptance) must drive **that** path (e.g. POST the start endpoint, then assert the loop responds), or **explicitly register the gap**. Echoes the ColonyGame lesson: verify the path the human actually takes, not just the path that happens to auto-run. *(confidence: high)*
