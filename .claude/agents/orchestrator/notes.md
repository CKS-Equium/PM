# Orchestrator — notes

Playbook for the Orchestrator. This agent appends freely; entries are **not** gated.
Entry format: **situation → what happened → lesson → confidence**.
The Process Engineer periodically distills proven lessons here into `persona.md` (via reviewed PR).

## team-pulse-dashboard (2026-06-03)

- **Live discovery interview** — *Situation:* gate-1 discovery for team-pulse-dashboard. *What happened:* a PM subagent cannot conduct a live, back-and-forth interview with the operator (no interactive channel), so I ran the interview turn-by-turn in the main thread, then handed the transcript to the PM to draft the PRD. *Lesson:* keep the live operator interview in the Orchestrator's main thread; the PM consumes the captured Q&A and authors the PRD. Do not delegate the live interview itself. *(confidence: high)*
- **Gate-1-before-repo ordering** — *Situation:* start-project created the project repo and registry before the PRD was approved. *What happened:* the PRD took 2 rounds (operator reframed to a live activity log + "Needs-you" panel); had it been rejected we'd have an orphan repo. *Lesson:* sequence start-project so gate-1 PRD approval precedes `gh repo create`; no project repo until we know we're building the right thing. *(confidence: high)*
- **New-project directory registration** — *Situation:* sub-agents' Glob/Read against the freshly-created project repo triggered directory-access prompts because the path wasn't in `additionalDirectories`. *What happened:* I patched it by adding `d:\github` to settings; fragile and manual. *Lesson:* start-project should auto-register each new project dir into `settings.local.json` so spawned agents have access without prompts. *(confidence: high)*
- **GitHub multi-issue auto-close** — *Situation:* PR #15 body used `Closes #5, #6, …` to close ten tickets on merge. *What happened:* GitHub only auto-closed #5 (one keyword binds one issue); the rest were closed manually. *Lesson:* use a closing keyword per issue (`Closes #5, Closes #6, …`) or close on merge programmatically. *(confidence: high)*

## hearthflix (2026-06-04)

- **Our own process fixes paid off (positive reinforcement)** — *Situation:* second project, first run of the post-team-pulse start-project changes (gate-1-before-repo, live interview in main thread, project-dir auto-registration). *What happened:* gate 1 passed **first try**, no orphan repo was created, and there were **zero directory-access prompts** for spawned agents across the whole run. *Lesson:* the recursive-improvement loop is working — recommendations applied at gate 7 measurably removed the exact friction they targeted. Keep treating prior post-mortem fixes as load-bearing and verify them on the next run. *(confidence: high)*
- **Multi-issue auto-close STILL unreliable — supersede the keyword fix** — *Situation:* applied team-pulse's "one `Closes #N` keyword per issue" fix on PR #17 (13 tickets). *What happened:* only ~6 of 13 issues auto-closed on merge; the rest were closed manually again. The keyword-per-issue convention is not robust at scale. *Lesson:* do **not** rely on PR-body keywords for closing at all — after merge, close tickets **programmatically in a loop** (`gh issue close` over the ticket list). This supersedes team-pulse rec #4. *(confidence: high)*
- **Narration for no-Bash agents** — *Situation:* product-manager, researcher-analyst, ux/ui-designer have no Bash/gh tool, so they cannot comment on their own issues to self-narrate. *What happened:* I narrated on their behalf for the activity log; works but conflates authorship and is easy to forget. *Lesson:* either grant a scoped narration capability to those roles or formalize "Orchestrator narrates for no-Bash agents" as an explicit convention. Noted at team-pulse, still unresolved — escalate as a process decision. *(confidence: med)*

## colonygame / Hiraeth (2026-06-08) — execution-mode autonomous build

- **Execution MODE: per-gate project DoD + reviewer-as-gate replaces the 7-gate lifecycle; BLOCK = fix-before-merge worked** — *Situation:* the operator pointed the team at their **own** Godot/C# colony-sim with its own `docs/DEV_PLAN.md` and a T0→T5 gate ladder, and directed: execute the plan (no discovery/design), use PM **PR conventions** (branch `gate/tN` + PR per gate, **self-merge on green DoD**), **don't stop at internal human gates**, run until plan-complete-needs-playtest or a hard blocker. *What happened:* I drove the whole overnight run with the **project's per-gate DoD** (`dotnet build *.sln` + sim `dotnet test` + headless `--scenario=tN` asserting end-state + Godot `--headless --import` + a `grep "using Godot" src/Sim` guard) **replacing** the 7-gate lifecycle, the **reviewer as the review gate**, and a reviewer **BLOCK treated as fix-before-merge, not a human stop**. I did **independent DoD verification** per gate (didn't take the engineer's word). Result: six clean gates merged, **0 escaped defects**, run stopped exactly at plan-complete-needs-playtest. *Lesson:* this is a valid first-class **alternative lifecycle** for operator-designed, no-discovery "execute the plan" runs — capture it (DESIGN/gates.md) so the next run starts from the proven bar; **BLOCK = fix-before-merge** keeps quality high without human stops; always do independent per-gate DoD verification yourself. *(confidence: high)*
- **The build-DoD must build the FULL solution, not just the sim assembly** — *Situation:* the T2 gate's DoD includes "scene imports & runs." *What happened:* the engineer ran `dotnet test` on the sim only, so a `Vec2` `using` error in the **Godot project** slipped through and the old binary ran "unknown scenario" — the "scene imports & runs" guarantee was hollow. I fixed it by requiring `dotnet build ColonyGame.sln` (whole solution) per gate. *Lesson:* when verifying a gate DoD, build the **entire solution** (incl. the framework/Godot project), not just the unit-tested assembly — a passing `dotnet test` on one project does not prove the shipping artifact compiles. **Candidate build-DoD.** *(confidence: high)*
- **Godot+C# in an autonomous loop needs a generated-artifact commit policy + post-merge sync recipe** — *Situation:* every headless run regenerates Godot `.uid`/`.godot`/`.import` artifacts. *What happened:* those dirtied the working tree and **blocked the post-merge local-`main` fast-forward** each gate, forcing a manual `stash -u → ff → drop` dance ×6. *Lesson:* before an autonomous Godot+C# run, define an upfront **`.uid`/generated-artifact commit-or-ignore policy** and a **defined post-merge sync recipe**, so the fast-forward after each self-merge is clean instead of needing a manual stash dance. *(confidence: high)*

## colonygame / Hiraeth (2026-06-08) — DEV_PLAN_2 player-first QoL (playtest-gated)

- **For player-facing work the human playtest is THE gate; green machine-DoD is necessary-but-insufficient — gate the playtest request behind a pre-playtest interaction review** — *Situation:* the QoL phase where the gate shifted from a headless assertion to a human playtest (operator plays from their iPad, replies in a `needs-human` dashboard issue, I merge on "passes"). *What happened:* I drove the cadence cleanly (build + machine-verify → raise playtest issue → merge on pass / fix-and-resubmit on fail — **no failing playtest was ever merged**), but I had **no instrument to catch the GUI bug class before the operator did**, so the **iteration count was high**: build & grow flow took **3 playtest rounds**, P1b took 2, and "menu not clickable" reached the operator **twice**. I treated green machine-DoD (`dotnet build` + 234 tests + `t0–t5` green) as "ready to playtest" when for interactive work it only means "ready to *attempt* a playtest." *Lesson:* for player-facing phases, **green machine-DoD ≠ playtest-ready** — read it as "the sim didn't regress," not "the game works." Gate the playtest request behind a **GUI interaction self-audit** (engineer) and a **pre-playtest interaction review** (reviewer reads the GUI diff for the known pitfalls: controls rebuilt per frame, UI enable-state not mirroring sim preconditions, input/movement coupled to the sim loop, per-frame allocs / always-on-top draws, dev affordances in the player build) to cut the operator round-trip count; expect game-feel iteration regardless. *(confidence: high)*
- **The dashboard `needs-human` playtest loop + fix-and-resubmit discipline is the right operator-interaction mechanism — POSITIVE** — *Situation:* the operator interacts asynchronously from their iPad and can't watch the run. *What happened:* across **6 playtest issues** (#8/#10/#12/#14/#16/#18) the operator played each phase, replied in-issue, and I merged on "passes" or fixed-and-resubmitted on fail; the loop matched exactly how the operator works and **fix-and-resubmit discipline held throughout** (no failing playtest merged). *Lesson:* the dashboard `needs-human` issue loop is the proven human-gate for player-facing autonomous work — keep it; pair it with the pre-playtest interaction review (above) to lower the round count. *(confidence: high)*

## new-cadair A/B experiment (2026-06-16) — autonomous Phase-A rebuild vs reference

- **A self-contained one-command verifier lets the orchestrator integrate a sub-phase whose building agent died — re-verify from artifacts, don't resurrect context** — *Situation:* the final sub-phase (A.6: CI + system-validation gate) of an autonomous Phase-A build; the delegated engineer **died twice on transient API errors** (500, then 529) mid-build, returning no final report. *What happened:* instead of trying to revive the dead agent, I inspected what it had left on the branch (a `ci.sh`, an updated evidence record, the flaky-test fix), **ran `ci.sh` myself** (7/7 stages green, 158 tests, then the full suite 5× consecutively → 0 flakes to prove determinism), **read the evidence doc for honesty** (confirmed the 12 deferrals were named/routed, not hidden gaps), then committed + merged it as PR #6. No agent-context resurrection needed. *Lesson:* when a build sub-phase ships a **one-command verifier + an honest evidence record**, the orchestrator can independently re-verify and integrate after an agent dies — make that the standing recovery path for a delegate that fails on a transient/terminal error mid-sub-phase, rather than re-running the whole delegation. Pairs with the process-engineer's "evidence-as-resilience" note. *(confidence: high)*
