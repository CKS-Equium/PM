# Senior Software Engineer — notes

Playbook for the Senior Software Engineer. This agent appends freely; entries are **not** gated.
Entry format: **situation → what happened → lesson → confidence**.
The Process Engineer periodically distills proven lessons here into `persona.md` (via reviewed PR).

## team-pulse-dashboard (2026-06-03)

- **Sanitize external-data hrefs at render** — *Situation:* I built the dashboard rendering links from GitHub data; junior was not spawned, I built directly. *What happened:* gate 5 found a HIGH `javascript:` XSS sink (no scheme check), missing CSP/headers, and a non-loopback bind. I fixed with a `safeUrl` https-allowlist, CSP/security headers, and loopback binding. *Lesson:* default to a scheme allowlist for any href/URL built from external data, add CSP, and bind local tools to loopback — do this on first write, not after review. *(confidence: high)*
- **Resolve derived state from the right subset** — *Situation:* milestone/phase was derived from issues. *What happened:* I derived it from an arbitrary/closed issue, producing a false phase-conflict badge; also active-role was hardcoded to "orchestrator" and an in-progress event class was silently empty. *Lesson:* when deriving state from a collection, filter to the meaningful subset (open issues only), drive role from the role label, and never let an event class render silently empty — assert it has content or surface "none". *(confidence: high)*
- **Normalize shared identifiers at the boundary** — *Situation:* registry gave `repo` as a URL; architecture examples used `owner/name`. *What happened:* I normalized at the ingestion boundary, which worked. *Lesson:* when contract and source artifact disagree on identifier shape, normalize once at the boundary and keep one canonical form internally. *(confidence: med)*

## hearthflix (2026-06-04)

- **Secure-by-default at first write — RECURRING, didn't transfer** — *Situation:* I built the FastAPI/Docker app. Despite my team-pulse note "add CSP/headers/loopback on first write," gate 5 again surfaced security blockers: no security headers/CSP (FIND-01), CVE-bearing deps (jinja2 3.1.5, python-multipart — FIND-02), Docker running as **root** (FIND-03), bind to `0.0.0.0` (FIND-04). I fixed all of them after review. *What happened:* the lesson existed in notes but did not become a first-write habit — the same class of finding recurred across two projects. *Lesson:* run a fixed **secure-by-default build checklist before requesting review**: (1) security headers + strict CSP; (2) dependency audit / pin to non-CVE versions; (3) non-root container user; (4) configurable bind, default loopback. This is now proven necessary twice — strong candidate to promote into the build DoD / my contract so it can't be skipped. *(confidence: high)*
- **No wait-state may be unbounded — poll liveness** — *Situation:* the transcode `ready()` check waited for the HLS playlist to appear before serving. *What happened:* if ffmpeg died immediately, `ready()` ignored `process.poll()` and the UI sat "Preparing forever" (M1). *Lesson:* any wait on an external process/resource must also watch for that dependency's **death/failure** (poll process liveness, timeout) and transition to an explicit error state — never let a wait be unbounded with only a success exit. *(confidence: high)*
- **Own the lifecycle of resources you create** — *Situation:* per-session HLS segment dirs under `/data`. *What happened:* I relied on ffmpeg's `delete_segments` (a no-op in `event` mode) to bound disk; it didn't, so disk grew unbounded (H1). I fixed it with explicit session-dir teardown on stop. *Lesson:* if I create a resource (temp dir, process, file), I own deleting it explicitly on teardown — don't trust an external tool's incidental cleanup to enforce a bound, and verify the bound holds. *(confidence: high)*

## colonygame / Hiraeth (2026-06-08) — execution-mode autonomous build

- **Guard authored-data correctness: explicit discriminators + load-time validation — RECURRING ROOT** — *Situation:* a data-driven Godot/C# colony-sim where building behaviour comes from `data/buildings.json`. *What happened:* at T1 the smelter-vs-mine kind was **inferred from an overloaded `reach` field**, so a mis-authored `reach` made the smelter silently never smelt — the economy didn't close, and it was caught **only by the headless scenario, not the 38 unit tests**. The same class recurred (T2 untested distinct-units, T4 warehouse stranding). I fixed it with an explicit `ProductionKind` discriminator + **load-time validation that throws `InvalidDataException` naming the offending building** on bad data. *Lesson:* never infer a behavioural kind from an incidental/overloaded field — use an **explicit discriminator** and **validate authored data at load time, throwing on bad data** so a mis-authored file fails loudly instead of silently producing nothing. This was the dominant recurring root of the whole run — strong candidate for the build DoD / my contract. *(confidence: high)*
- **Acceptance scenarios must exercise the full produce→consume loop (assert usability, not presence/conservation)** — *Situation:* the T4 logistics gate added warehouses + couriers, verified by a headless scenario. *What happened:* warehouses were a **one-way "roach motel"** — couriers *deposited* production goods but no phase *read them back*, silently draining steel/circuits into dead storage. The scenario **masked** it by warehousing a throwaway good nothing consumes, and the tests asserted only **conservation** (goods weren't lost — they were just stuck), never **usability**. The reviewer caught it; I fixed with a round-trip retrieval pass + a regression test (15 stranded without the fix → 0 with). *Lesson:* an acceptance scenario for any storage/transport/transform stage must drive the **full produce→consume loop** and assert the output is **actually usable** downstream — "conserved" and "present" are not "consumable." Don't seed throwaway goods that hide the real path. *(confidence: high)*
- **Verify the FULL-solution build every gate, not just the sim assembly** — *Situation:* at T2 I wrote the scenario and ran `dotnet test` on the sim, then handed off. *What happened:* I built only the sim assembly, so a `Vec2` `using` error in the **Godot project** slipped through — the old binary ran "unknown scenario" and the gate's "scene imports & runs" guarantee was never actually proven. *Lesson:* the per-gate DoD's "scene imports & runs" is only real if I `dotnet build <Solution>.sln` (the **whole** solution incl. the Godot project) before requesting review — `dotnet test` on one assembly is not a build of the thing that ships. *(confidence: high)*
- **The pure-(non-Godot) sim seam made an unverifiable game autonomously buildable — POSITIVE** — *Situation:* a Godot game normally needs a human to judge "does it work." *What happened:* keeping `src/Sim` **Godot-free** (command-in/state-out, enforced by a `grep "using Godot" src/Sim` guard) plus a headless `--scenario=tN` runner that asserts end-state and prints `[GATE tN PASS]`/exit code let me build, test, and gate the whole game with no human in the loop — and the scenario caught the T1 economy bug unit tests missed. *Lesson:* for any game/UI-heavy build, push the logic behind a **headless, framework-free seam** and write a **scenario runner that asserts end-state** as the machine proxy for "is it playable"; it is the single thing that made the autonomous run possible. *(confidence: high)*

## colonygame / Hiraeth (2026-06-08) — DEV_PLAN_2 player-first QoL (playtest-gated)

- **GUI/interaction CANNOT be headless-verified — run a GUI interaction self-audit before any playtest — RECURRING (within-phase)** — *Situation:* the player-first QoL phase where the gate was a human playtest, not a headless assertion; I (and the orchestrator) are both headless and cannot click-test. *What happened:* the machine-DoD (`dotnet build` + 200+→234 tests + the `t0–t5` scenarios) stayed **green while the game was broken in the hand**, and a whole bug class surfaced *only* at the operator's playtest — repeatedly: (1) the **build menu rebuilt every Button every frame**, so a click's press→release landed on a button destroyed mid-click, `Pressed` never fired — "menu not clickable" was reported **TWICE** before I found it (number-keys masked it); (2) the **Recruit button silently no-op'd** — it looked enabled but the sim refused (lone vagrant unfed → avg-sat 0) because the button's enable-check **omitted the sim's satisfaction precondition** (a lying button); (3) the fixed-timestep refactor **buried input polling + movement inside the 20Hz sim loop** and rendered the player with no interpolation → at 119fps the avatar snapped 20×/s and **fast clicks were dropped** + movement was choppy; (4) couldn't click the Player Home because the vagrant **rendered exactly on top of it** (pick/co-location); (5) **FPS tanked** from always-on-top `NoDepthTest` `Label3D` billboards on every node+building + per-frame string churn; (6) leftover **dev/sandbox keys** (`B`/`1–4`/`R`/`F`) leaked into the player build, placing free unlabeled buildings (key `3` placed the *staffed* T1 smelter, not the player-fed one → "built smelter, 100 ore+coal, no output"). *Lesson:* none of these are catchable headless — before requesting a playtest, run a written **GUI interaction self-audit**: (a) **never rebuild interactive controls every frame** — persist them, rebuild only on tab/roster change; (b) **capture input every frame**, never inside a fixed-timestep loop; (c) **interpolate rendered positions between sim ticks**; (d) avoid always-on-top/`NoDepthTest` labels + per-frame allocations at scale; (e) **UI enable-state must mirror the sim's exact preconditions** — a silently-no-op'ing button is a bug; (f) **never ship dev/sandbox affordances in the player build**. This is the dominant recurring root of the whole phase — strong candidate for the player-facing build-DoD / my contract. *(confidence: high)*
- **The decoupled Godot-free sim core + the `t0–t5` regression net held through ~9 UX PRs — POSITIVE** — *Situation:* a phase of heavy presentation/UX churn (cursor-picking, a tabbed build menu, a fixed-timestep refactor, `FounderTier`/`PlayerTier` reconciliation) on top of the proven sim. *What happened:* **~9 PRs of UX work rode on the sacred sim core with 0 accidental sim regressions** — the `t0–t5` headless scenarios never broke once and the T5 win stayed intact. *Lesson:* the sacred-sim seam + a kept-green regression net is what lets you iterate fast and safely on the surface of a UI-heavy build; keep the seam and never let surface work mutate the sim. Read green machine-DoD as **"the sim didn't regress," not "the game works"** — the feel/UI gate is entirely human. *(confidence: high)*
